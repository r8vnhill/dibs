## üì¶ Registros con `dataclass`

Desde la versi√≥n 3.7, Python incluye el decorador [`@dataclass`](https://docs.python.org/3/library/dataclasses.html), que permite declarar clases centradas en datos con una sintaxis concisa y funcionalidades generadas autom√°ticamente.

Un registro t√≠pico en Python con `dataclass` declara sus atributos directamente en la clase. Al igual que en Kotlin, el decorador se encarga de generar:

- `__init__` para inicializar los atributos.

- `__eq__` para comparar por contenido.

:::warning Mutabilidad por defecto

Los campos de una `dataclass` son **mutables por defecto**, lo que significa que sus valores pueden cambiarse despu√©s de la creaci√≥n del objeto.

:::

:::danger Valores por defecto

### ‚öñÔ∏è Comparaci√≥n por contenido

En Python, las clases **comunes** comparan por **referencia**, al igual que en Kotlin.  
Sin embargo, al usar `@dataclass`, se genera autom√°ticamente un m√©todo `__eq__`, lo que permite comparar **por contenido**.

```python title="Clase com√∫n: comparaci√≥n por referencia" showLineNumbers

    model: str
    power: int

    def __init__(self, model: str, power: int):
        self.model = model
        self.power = power

if __name__ == '__main__':
    mark1 = Armor("Mark I", 75)
    mark1_clone = Armor("Mark I", 75)
    print(mark1 == mark1_clone) # False
```

```python title="@dataclass: comparaci√≥n por contenido" showLineNumbers
@dataclass
class Armor:
    model: str
    power: int


if __name__ == '__main__':
    mark2 = Armor("Mark II", 100)
    mark2_clone = Armor("Mark II", 100)
    print(mark2 == mark2_clone) # True
```

<Explanation>
    Al igual que en Kotlin, el decorador `@dataclass` genera autom√°ticamente el m√©todo `__eq__` en Python,  
    lo que permite comparar las instancias por el valor de sus atributos en lugar de su identidad en memoria.

    - Las clases comunes heredan el comportamiento por defecto de `object`, que compara por referencia ‚Üí `False`.
    - Las `dataclass` implementan comparaci√≥n por contenido ‚Üí `True`.

    Esta caracter√≠stica es esencial cuando se usan las instancias como representaciones de datos (tipos producto),

</Explanation>

<details>

<summary>üî¨ ¬øY si quiero comparar por referencia en Python?</summary>

Puedes usar el operador `is`, que verifica si **dos variables apuntan al mismo objeto** en memoria:

```python

```

:::tip

Al igual que en Kotlin, usa `is` solo cuando te interese verificar identidad.  
Para estructuras de datos y tipos producto, la comparaci√≥n por contenido (`==`) es lo m√°s com√∫n y √∫til.

:::

</details>

## üéØ Conclusiones

En esta lecci√≥n comparamos las `data class` de Kotlin con las `@dataclass` de Python, explorando c√≥mo ambos lenguajes ofrecen mecanismos similares para representar tipos centrados en datos. Tambi√©n destacamos el impacto de estas construcciones en la **comparaci√≥n de objetos**: mientras las clases comunes comparan por referencia, tanto en Kotlin como en Python, los mecanismos de clases de datos permiten **comparar por contenido**, lo cual es fundamental al trabajar con tipos producto.

### üîë Puntos clave

- Python ofrece `@dataclass` desde la versi√≥n 3.7 como una forma concisa de definir registros.
- Al igual que en Kotlin, estas clases generan autom√°ticamente m√©todos como `__init__`, `__repr__` y `__eq__`.
- Los campos son mutables por defecto, pero se puede cambiar a inmutabilidad usando `frozen=True`.
- Es importante definir primero los campos sin valor por defecto.
- Las `dataclass` comparan por contenido (`==`), mientras que las clases comunes lo hacen por referencia, salvo que se implemente `__eq__`.
- Para comparar por referencia en Python, se usa el operador `is`, equivalente a `===` en Kotlin.
