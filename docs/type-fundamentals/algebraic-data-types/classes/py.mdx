### üèóÔ∏è Declaraci√≥n de clases

En Python, una clase se define con la palabra clave `class`, y sus propiedades se inicializan dentro del m√©todo especial `__init__`, que act√∫a como constructor. Las anotaciones de tipo se declaran usando `:` despu√©s del nombre del atributo o par√°metro:

```python showLineNumbers title="Clase b√°sica en Python (type-fundamentals/algebraic_types/product/classes.py)"
class Point:
    x: int
    y: int

    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y
```

<Explanation>
    Declaramos una clase `Point` con dos propiedades (`x` e `y`), ambas de tipo `int`. Las anotaciones de tipo aparecen tanto en la declaraci√≥n de atributos como en los par√°metros del constructor `__init__`.
</Explanation>

Este patr√≥n cumple el mismo rol que el **constructor primario** en Kotlin: permite inicializar los valores que componen el tipo. Sin embargo, en Python es necesario:

- Anotar los tipos por separado.
- Declarar expl√≠citamente las propiedades fuera del constructor.
- Asignarlas manualmente dentro del cuerpo del `__init__`.

En Kotlin, en cambio, estas acciones se combinan en una sola l√≠nea, lo que da lugar a una **sintaxis m√°s declarativa, compacta y libre de redundancia**.

::::info `self`

- `self` es una convenci√≥n en Python que representa la instancia actual de la clase. Es equivalente a `this` en otros lenguajes como Kotlin.
- Todo m√©todo de instancia debe tener `self` como primer par√°metro, lo que permite acceder a atributos y otros m√©todos de la instancia.

:::warning Error com√∫n

Un error frecuente es **olvidar incluir `self` como primer par√°metro** en el m√©todo `__init__`, lo que hace que Python malinterprete los argumentos al instanciar la clase.

```python showLineNumbers title="path/to/file"
class Point:
    x: int
    y: int

    def __init__(x: int, y: int):  # ‚ùå Falta self
        self.x = x
        self.y = y
```

Esto genera el siguiente error en tiempo de ejecuci√≥n:

```plaintext
Traceback (most recent call last):
  File "path/to/file", line 38, in <module>
    pos: Final[Point] = Point(1, 2)
                        ^^^^^^^^^^^
TypeError: Point.__init__() takes 2 positional arguments but 3 were given
```

El problema es que Python espera que el primer par√°metro de un m√©todo de instancia sea una referencia a la propia instancia. Como no escribimos `self`, el nombre `x` ocupa ese lugar, por lo que al llamar `Point(1, 2)`, Python interpreta que est√°s pasando **tres** argumentos: `self`, `1` y `2`, cuando la funci√≥n solo espera dos.

<details>
<summary>Abusando de la sintaxis: `this`</summary>

Dado que `self` es solo una convenci√≥n, **podr√≠amos usar cualquier otro nombre**, pero **no es recomendable**. Por ejemplo:

```python showLineNumbers title="Abusando de la sintaxis"
class Point:
    x: int
    y: int

    def __init__(this, x: int, y: int):
        this.x = x
        this.y = y
```

Esto funcionar√° correctamente, pero **rompe la legibilidad y las convenciones del lenguaje**, por lo que debe evitarse.

</details>

## üß™ Instanciaci√≥n

Instanciar una clase en Python es similar a Kotlin en cuanto a sintaxis: se omite el uso de `new`, y los argumentos se pasan directamente al constructor.

```python showLineNumbers title="Instanciaci√≥n de una clase (type-fundamentals/algebraic_types/product/classes.py)"
point: Final[Point] = Point(49, 41)
print(point.x)  # 49
print(point.y)  # 41
```

<Explanation>
    Creamos una instancia de `Point` pasando los valores `49` y `41` al constructor. La variable `pos` est√° anotada con `Final`, lo que indica que **no debe ser reasignada**. Esto se asemeja a `val` en Kotlin, aunque en Python esta restricci√≥n no es estricta en tiempo de ejecuci√≥n, sino solo una **sugerencia para herramientas de an√°lisis est√°tico** como `mypy` o `ty`.
</Explanation>

## üß± Encapsulamiento y l√≥gica interna

Python permite incluir m√©todos en una clase igual que en Kotlin. Aqu√≠ usamos **atributos privados**, **m√©todos**, y **propiedades** para encapsular la l√≥gica:

<BoxedTabs>

<TabItem label={"C√≥digo esencial"} value={"C√≥digo esencial"}> {/*  region : C√ìDIGO ESENCIAL */}

```python showLineNumbers title="Clase con l√≥gica interna (type-fundamentals/algebraic_types/product/classes.py)"
class Position:
    def __init__(self, x: int, y: int):
        self.__x = x
        self.__y = y

    def move(self, dx: int, dy: int) -> 'Position':  # 'Position' es una forward reference
        return Position(self.__x + dx, self.__y + dy)

    @property
    def is_origin(self) -> bool:
        return self.__x == 0 and self.__y == 0
```

</TabItem>  {/* endregion C√ìDIGO ESENCIAL */}

<TabItem label={"C√≥digo completo"} value={"C√≥digo completo"}>   {/*  region : C√ìDIGO COMPLETO */}

```python showLineNumbers title="some title (and path/to/file)"

__x: int
__y: int

def __init__(self, x: int, y: int):
    self.__x = x
    self.__y = y

def move(self, dx: int, dy: int) -> 'Position':
    return Position(self.__x + dx, self.__y + dy)

@property

@property

@property
```

```python showLineNumbers title="some title (and path/to/file)"

print(position.move(-86, -29).is_origin)    # True
```
</TabItem>  {/* endregion C√ìDIGO COMPLETO */}

</BoxedTabs>

<Explanation>
    En este ejemplo:
    
    - Usamos `__x` y `__y` con doble guion bajo para indicar que son **atributos privados**, siguiendo una convenci√≥n de ocultamiento.
    - El m√©todo `move` crea una **nueva instancia** con los valores actualizados, siguiendo un estilo inmutable.
    - Usamos `@property` para exponer `x`, `y` e `is_origin` como **atributos de solo lectura**. Esto permite acceder a ellos como si fueran campos (`pos.x`), sin par√©ntesis.
        - Ya que no definimos un setter, estos atributos son **inmutables** desde el exterior.
    - La anotaci√≥n `'Position'` en `-> 'Position'` es una **forward reference**, necesaria porque estamos usando el tipo dentro de su propia definici√≥n.
</Explanation>

<details>
<summary>üîì Abusando de la sintaxis: Rompiendo el encapsulamiento</summary>

```python showLineNumbers title="Intento 1: setter en propiedad de solo lectura"
position: Final[Position] = Position(45, 64)
position.x = 85
```

Esto generar√° un error en tiempo de ejecuci√≥n, ya que `x` es una **propiedad de solo lectura**:

```plaintext
Traceback (most recent call last):
  File "path/to/file", line 42, in <module>
    position.x = 85
    ^^^^^^^^^^
AttributeError: property 'x' of 'Position' object has no setter
```

```python showLineNumbers title="Intento 2: asignaci√≥n directa a atributo 'privado'"
position: Final[Position] = Position(45, 64)
position.__x = 85
print(position.x)  # 45
```

Este c√≥digo no lanza error, pero **no modifica** el valor real del atributo `__x`. Lo que sucede es que se crea un nuevo atributo llamado `__x` en la instancia, **distinto** del atributo interno original.

```python showLineNumbers title="Intento 3: name mangling para forzar el acceso"
position: Final[Position] = Position(45, 64)
position._Position__x = 85
print(position.x)  # 85
```

Aqu√≠ s√≠ se modifica el valor interno real, accediendo al atributo a trav√©s de su nombre transformado por *name mangling*.

---

En realidad, Python no impide el acceso a atributos privados. Lo que hace es aplicar **[name mangling](https://commons.wikimedia.org/wiki/File:Mangle.jpg#/media/File:Mangle.jpg)** (renombrado de atributos), anteponiendo el nombre de la clase al atributo (por ejemplo, `__x` se convierte en `_Position__x`). Esto **no garantiza encapsulamiento real**, sino que act√∫a como una convenci√≥n para evitar colisiones accidentales y desalentar el acceso directo.

</details>

## üèó ¬øConstructores secundarios?

En Python **no existe el concepto de constructores primario y secundarios** como en Kotlin. En lugar de eso, se suelen usar otros patrones y mecanismos que permiten lograr comportamientos similares.

A continuaci√≥n, se presentan algunas formas comunes de definir m√∫ltiples maneras de crear instancias de una clase:

### üè≠ M√©todos de f√°brica est√°ticos

Son funciones que devuelven instancias de la clase y permiten tener distintos nombres y l√≥gicas para crear objetos.

### ‚ú® Uso de `*args` y `**kwargs`: Inicializaci√≥n flexible

En Python, `*args` y `**kwargs` permiten crear constructores **flexibles y gen√©ricos**, aceptando m√∫ltiples formas de inicializar una clase sin necesidad de definir varios constructores como en Kotlin.

Esto es √∫til cuando:

- Los par√°metros pueden venir en diferentes formas (tuplas, diccionarios, etc.).
- Se desea simular **constructores secundarios** sin replicar c√≥digo.
- Se busca compatibilidad con otras APIs o configuraciones din√°micas.

Supongamos que estamos implementando una clase `DenseLayer`, inspirada en frameworks de deep learning:

```python showLineNumbers title="Clase con inicializaci√≥n flexible usando *args y **kwargs (type-fundamentals/algebraic_types/product/classes.py)"
class DenseLayer:
    __input_dim: int
    __output_dim: int
    __activation: str
    __use_bias: bool
    
    def __init__(self, *args: int, **kwargs: dict[str, int | str | bool]):
        if args:

        }
```

### üß± Builder Pattern: Simulando constructores m√∫ltiples

En Python, si una clase tiene muchos par√°metros opcionales o configuraciones posibles, una buena forma de simular **constructores secundarios** (como en Kotlin) es usar el **patr√≥n Builder**. Este patr√≥n te permite construir objetos paso a paso, manteniendo el constructor de la clase principal limpio y enfocado.

#### Ejemplo: Creaci√≥n flexible de una clase `User`

```python showLineNumbers title="user_builder.py"
class User:
    def __init__(self, name: str, age: int, email: str | None = None, phone: str | None = None):
        self.name = name
        self.age = age
        self.email = email
        self.phone = phone

class UserBuilder:
    def __init__(self, name: str, age: int):
        self._name = name
        self._age = age
        self._email = None
        self._phone = None

    def with_email(self, email: str) -> 'UserBuilder':
        self._email = email
        return self

    def with_phone(self, phone: str) -> 'UserBuilder':
        self._phone = phone
        return self

    def build(self) -> User:
        return User(
            name=self._name,
            age=self._age,
            email=self._email,
            phone=self._phone
        )
```

#### Uso del builder

```python showLineNumbers title="example.py"
user1 = UserBuilder("Ana", 28).with_email("ana@example.com").build()
user2 = UserBuilder("Luis", 35).with_phone("555-1234").build()

print(user1.email)  # ana@example.com
print(user2.phone)  # 555-1234
```

---

Este enfoque tiene varias ventajas:

- Separa la l√≥gica de construcci√≥n de la clase principal.
- Permite una API m√°s fluida y legible.
- Es √∫til cuando hay m√∫ltiples formas v√°lidas de construir el objeto, simulando **constructores secundarios nombrados** como en Kotlin.

**Nota:** en Python, este patr√≥n es menos com√∫n gracias a `@classmethod`, `*args`, `**kwargs` y valores por defecto, pero sigue siendo √∫til en casos donde la construcci√≥n es compleja o quieres una API m√°s expresiva.

## üîç ¬øQu√© se puede hacer en Kotlin que no en Python?

* **Bloques `init`**: No hay equivalente directo en Python para ejecutar c√≥digo inmediatamente despu√©s de la construcci√≥n. En Python, la l√≥gica de validaci√≥n debe colocarse dentro de `__init__`.
* **Valores por defecto y propiedades en el encabezado de clase**: Kotlin permite declarar y asignar valores por defecto en el constructor primario, algo que en Python se debe hacer dentro del `__init__`, lo que genera m√°s repetici√≥n.
* **Visibilidad por modificadores (`private`, `internal`)**: Python no tiene control de visibilidad real. Usos como `_atributo` o `__atributo` son solo convenciones, no restricciones del compilador.

## üéØ Conclusiones

Tanto Kotlin como Python permiten modelar tipos producto con clases, encapsulando datos y operaciones. Sin embargo, **Kotlin ofrece una sintaxis m√°s concisa, declarativa y segura**, dise√±ada espec√≠ficamente para evitar errores comunes al modelar estructuras de datos.

Python, en cambio, ofrece m√°s flexibilidad y menos restricciones, pero requiere mayor disciplina del programador para lograr un dise√±o claro, seguro y mantenible.

Aprender las diferencias entre estos enfoques no solo ayuda a escribir mejor c√≥digo en cada lenguaje, sino tambi√©n a **elegir conscientemente c√≥mo representar nuestros dominios y dise√±ar nuestras bibliotecas.**
