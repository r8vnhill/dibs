## üí• Mutabilidad por defecto

En Python, las clases decoradas con `@dataclass` son **mutables por defecto**. Esto significa que puedes modificar los atributos de una instancia incluso despu√©s de haberla creado:

```python showLineNumbers title="Mutabilidad por defecto"
@dataclass

    title: str
    publisher: str

comic = Comic("Batman: I Am Gotham", "DC")
comic.publisher = "Marvel"  # Esto es v√°lido
```

Este comportamiento puede ser √∫til en scripts peque√±os o estructuras ef√≠meras, pero **no es deseable** cuando se quiere representar un tipo producto **estable, inmutable y seguro**, como suele hacerse en bibliotecas reutilizables.

Para declarar un registro **inmutable** en Python, puedes usar el par√°metro `frozen=True`:

```python showLineNumbers title="Inmutabilidad con frozen=True (type-fundamentals/algebraic_types/product/data_classes/comic.py)"
@dataclass(frozen=True)
class Comic:
    title: str
    publisher: str


if __name__ == "__main__":
    comic = Comic(title="Batman: I Am Gotham", publisher="DC")
    comic.publisher = "Marvel"
```

<Explanation>
    En este ejemplo, al intentar modificar `publisher`, Python lanza un `FrozenInstanceError`. Esto ocurre porque usamos `@dataclass(frozen=True)`, que convierte cada campo en **de solo lectura** una vez creada la instancia.

    Este enfoque es equivalente a declarar todas las propiedades como `val` en Kotlin. Es recomendable cuando modelas datos que **no deben cambiar** una vez definidos, como registros de base de datos, configuraciones o valores calculados.
</Explanation>

:::warning Inmutabilidad en tiempo de ejecuci√≥n

El par√°metro `frozen=True` **no garantiza inmutabilidad en tiempo de compilaci√≥n**, como ocurre con `val` en Kotlin.  
En su lugar, aplica una restricci√≥n en **tiempo de ejecuci√≥n**, interceptando cualquier intento de modificar los atributos y lanzando un `FrozenInstanceError`.

:::

## ‚ôªÔ∏è Copias inmutables con `replace`

En Kotlin, puedes usar `copy()` para crear una nueva instancia de una `data class` con campos modificados, sin alterar el original.  
En Python, puedes lograr un efecto similar con `@dataclass(frozen=True)` y la funci√≥n auxiliar [`replace`](https://docs.python.org/3/library/dataclasses.html#dataclasses.replace).

Esto permite mantener la inmutabilidad y al mismo tiempo modelar **transformaciones de estado**.

<BoxedTabs>

<TabItem label={"C√≥digo esencial"} value={"C√≥digo esencial"}>

```python showLineNumbers title="Creaci√≥n de copias inmutables con replace (type-fundamentals/algebraic_types/product/data_classes/ghoul.py)"
@dataclass(frozen=True)

    name: str
    hunger: int
```

```python showLineNumbers title="Uso de replace (type-fundamentals/algebraic_types/product/data_classes/ghoul.py)"
nishio_before = Ghoul(name="Nishiki Nishio", hunger=77)
print(nishio_before)
print("Eating...")
nishio_after = replace(nishio_before, hunger=nishio_before.hunger - 10)
print(nishio_after)
```

</TabItem>

<TabItem label={"C√≥digo completo"} value={"C√≥digo completo"}>

```python showLineNumbers title="Creaci√≥n de copias inmutables con replace (type-fundamentals/algebraic_types/product/data_classes/ghoul.py)"

from typing import Final


@dataclass(frozen=True)

    name: str
    hunger: int


if __name__ == "__main__":
    nishio_before: Final[Ghoul] = Ghoul(name="Nishiki Nishio", hunger=77)
    print(nishio_before)
    print("Eating...")
    nishio_after: Final[Ghoul] = replace(nishio_before, hunger=nishio_before.hunger - 10)
    print(nishio_after)
```

</TabItem>

</BoxedTabs>

<Explanation>
    
En este ejemplo, usamos `replace(...)` para crear una nueva instancia de `Ghoul` con el campo `hunger` modificado.  
Dado que `Ghoul` es una `@dataclass(frozen=True)`, **no podemos modificar sus atributos directamente**, pero s√≠ podemos crear una copia con valores nuevos.

Este patr√≥n es equivalente al uso de `copy()` en Kotlin, y es clave para **mantener inmutabilidad sin sacrificar expresividad**.

</Explanation>

En Kotlin puedes desestructurar una `data class` directamente, gracias a los m√©todos `componentN()` generados autom√°ticamente.

<TabItem label={"C√≥digo esencial"} value={"C√≥digo esencial"}>

<TabItem label={"C√≥digo completo"} value={"C√≥digo completo"}>


</TabItem>

</BoxedTabs>

<Explanation>

Aqu√≠ usamos `astuple(...)` para convertir una instancia de `VideoGame` en una tupla como `("The Awesome Adventures of Captain Spirit", "Dontnod Entertainment")`.  
Esto nos permite desestructurar el objeto usando la sintaxis habitual de Python, aunque el lenguaje **no ofrece esta capacidad de forma autom√°tica** como s√≠ lo hace Kotlin con `componentN()`.

Si necesitas que la instancia de una `@dataclass` se desestructure sin usar `astuple`, puedes implementar el m√©todo especial `__iter__`, aunque eso rompe la sem√°ntica de tu tipo producto si no est√°s modelando una secuencia.

## üîÇ Funciones auxiliares y propiedades derivadas

## üõ°Ô∏è Validaci√≥n con `__post_init__`

Este patr√≥n es equivalente al uso de `require(...)` en el bloque `init` de Kotlin, y permite definir clases de datos **robustas y seguras**, que validan su propio estado.

## üéØ Conclusiones

A lo largo de esta lecci√≥n exploramos c√≥mo Python permite representar tipos producto de forma idiom√°tica usando `@dataclass`.  
Aunque el lenguaje no ofrece todas las garant√≠as de compilaci√≥n de Kotlin, provee mecanismos expresivos que cumplen un rol similar al modelar datos estructurados.

Tambi√©n contrastamos con Kotlin para identificar diferencias importantes en la sem√°ntica del lenguaje, especialmente en lo relacionado con inmutabilidad y transformaciones de estado.

### üîë Puntos clave

- Las `@dataclass` generan autom√°ticamente m√©todos como `__init__`, `__repr__` y `__eq__`, permitiendo representar tipos producto de forma concisa.

- Para modificar instancias inmutables, Python ofrece la funci√≥n `replace`, an√°loga a `copy()` en Kotlin.
- Python no ofrece desestructuraci√≥n autom√°tica, pero se puede lograr mediante `astuple` o definiendo `__iter__`.
- Las `@dataclass` pueden incluir funciones y propiedades derivadas, manteniendo la sem√°ntica de tipo producto sin necesidad de herencia ni l√≥gica compleja.
- El m√©todo `__post_init__` permite validar campos o inicializar valores derivados de manera segura, despu√©s de construido el objeto.

### üß∞ ¬øQu√© nos llevamos?

Al igual que en Kotlin, modelar tipos producto en Python usando `@dataclass` nos ayuda a escribir c√≥digo m√°s **legible, expresivo y f√°cil de mantener**.

Aprender a usar estas estructuras de forma idiom√°tica no solo mejora la calidad del c√≥digo, sino que te entrena para pensar en t√©rminos de **estructura y sem√°ntica**, no solo de implementaci√≥n.

Este enfoque es clave al dise√±ar **bibliotecas reutilizables**, donde los datos deben ser claros, validables, comparables y seguros por construcci√≥n.

Dominar `@dataclass` te permite construir modelos robustos, expresar reglas de dominio desde el tipo mismo, y mantener un estilo declarativo ‚Äîincluso en un lenguaje din√°mico como Python.
